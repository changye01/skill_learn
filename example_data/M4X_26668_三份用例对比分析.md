# M4X订单管理 26668 三份测试用例对比分析

## 概览


| 维度    | **v1** (72条)    | **v2** (39条)    | **aily** (30条) |
| ----- | --------------- | --------------- | -------------- |
| 场景分组  | 13组（含综合场景）      | 9组              | 无分组            |
| 用例编号  | TC-001 ~ TC-072 | TC-001 ~ TC-039 | TC001 ~ TC030  |
| 综合场景  | 有（4条）           | 无               | 有（3条）          |
| 超范围用例 | 无               | 无               | 有（性能/兼容性）      |


---

## v1 (72条) — 详细版

**文件：** `M4X_订单管理_测试用例_26668_v1.xlsx`

**优点：**

- 覆盖最全面，13个场景分组，粒度最细
- 订单状态按具体状态值拆分（已下载或已生成、订单处理中、取消、已完成等），贴合实际枚举值
- 权限验证拆到具体角色（产品调研人、产品开发人员分别验证），边界清晰
- 有"综合场景验证"组，覆盖端到端流程
- 字段完整性验证、账号含特殊字符等边界场景考虑到位

**缺点：**

- 用例偏多，部分用例有冗余（如"字段完整性验证"和"新增参数"重叠）
- 订单状态每个值一条用例，维护成本高，状态值变更时需批量改

---

## v2 (39条) — 精简版

**文件：** `M4X_订单管理_测试用例_26668_v2.xlsx`

**优点：**

- 精炼，每个需求点一条用例，不冗余
- 查询条件覆盖完整：精确、模糊、无匹配、清空、组合，实用性强
- 权限控制5条（3个有权限角色 + 2个无权限场景），覆盖到位不冗余
- 组合查询单独成例，符合实际测试习惯

**缺点：**

- 订单状态未按具体枚举值拆分，可能漏测某个状态的异常
- 缺少综合/端到端场景验证
- 缺少"已创建PO单后修改采购员"的影响场景
- 弹窗操作缺少"选择与当前相同采购员"的边界场景

---

## aily (30条) — 第三方AI生成

**文件：** `M4X_订单管理_测试用例_26668_aily.xlsx`

**优点：**

- 包含跨功能全流程验证（TC028），从同步→分配→创建PO的完整链路
- 有"数据一致性验证"（TC012），接口传入与列表展示对比
- 有"修改后数据持久化"验证（TC022），刷新页面确认数据不丢失

**缺点：**

- **无场景分组**，30条平铺，阅读和执行时需自行归类
- **有超范围用例**：TC029性能测试、TC030兼容性测试，这类通常不在功能测试范围内
- 步骤描述使用 `<br>` HTML标签而非换行符，Excel中展示不友好
- 部分用例过于粗略（如TC011"历史数据修复"一条覆盖M4X+DBS+准确性，颗粒度不够）
- 权限验证缺少"当前采购员修改他人订单"的场景
- 编号格式 TC001 与其他两份 TC-001 不统一

---

## 推荐建议


| 场景       | 推荐                                               |
| -------- | ------------------------------------------------ |
| 追求覆盖全面   | 用 **v1**，但可删减冗余的订单状态枚举用例（保留关键状态即可）               |
| 追求精简实用   | 用 **v2** 为底，补充 v1 的综合场景（4条）和 aily 的数据持久化验证       |
| **最佳方案** | **v2 基础 + v1 综合场景 + aily 数据一致性验证**，约45条，覆盖全面又不冗余 |
