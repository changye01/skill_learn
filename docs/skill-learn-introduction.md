# `skill-learn` 项目介绍

## 项目概述

`skill-learn` 是一个面向测试设计场景的 Claude Code Skills 插件集合，仓库地址为 [https://github.com/changye01/skill_learn](https://github.com/changye01/skill_learn)。

它当前聚焦解决两类高频问题：

- 把需求文档、接口说明、原型或已有材料，整理成可确认、可追溯的测试场景与结构化测试用例。
- 在测试用例已经确认的前提下，继续结合技术方案和 `reference-pack`，补齐可执行的测试数据与测试执行清单。

插件以单个 `skill-learn` 发布，当前包含两个核心 Skill：

- `test-case-generator`：先生成测试场景地图，再生成结构化测试用例。
- `test-data-generator`：在已确认用例基础上，继续生成测试执行清单与测试数据草稿。

## 它解决了什么问题

传统“需求转测试”的过程往往有三个痛点：

- 只直接写测试用例，缺少中间的“场景确认层”，容易漏测或误测。
- 测试数据设计常常脱离技术方案和库表结构，导致数据不可执行。
- AI 虽然能快速生成文档，但如果没有固定流程，容易出现格式不稳定、覆盖重复、关键规则丢失的问题。

`skill-learn` 的思路是把这件事拆成稳定的分阶段流程：

1. 先明确“测什么”。
2. 再明确“怎么测”。
3. 最后明确“拿什么数据去测”。

每一步都要求先对话确认，再落盘，并通过脚本校验 Markdown 结构完整性。

## 核心能力

### 1. `test-case-generator`

这个 Skill 负责从需求材料走到“确认版结构化测试用例”。

标准流程如下：

1. 读取需求文档与补充材料。
2. 拆出页面、接口、任务、权限、日志、历史修复等可测试对象。
3. 先输出 `测试场景地图` 草稿，和用户确认覆盖范围、待确认项、超范围项。
4. 用户确认后保存 `测试场景地图.md`。
5. 再生成 `结构化测试用例.md`。
6. 运行 `validate_cases.py` 校验 Markdown 结构。

它的关键特点不是“多写用例”，而是：

- 默认覆盖正常、边界、异常、权限、日志、数据一致性、数据持久化、综合链路。
- 对用户在对话中补充的明确规则，会沉淀到 `本次确认规则`。
- 对关键角色、关键枚举值默认展开，不用“有权限用户”“多种状态”这类模糊写法糊过去。
- 校验失败会先回改已保存文件，再重新校验。

### 2. `test-data-generator`

这个 Skill 负责从“确认版结构化测试用例”走到“可执行的测试执行清单”。

它不会把测试数据设计当成纯补表动作，而是明确要求三类输入：

- 已确认的 `结构化测试用例.md`
- 技术方案
- `reference-pack`（通常包含 `tables/*.sql` 与 `table_samples/*.csv`）

标准流程如下：

1. 继承基线用例，不随意改写测试目标。
2. 先读技术方案，确认字段映射、状态语义、权限规则、日志来源、历史修复逻辑。
3. 再读 `reference-pack`，确认落表字段与样例形态。
4. 按 `TC` 补齐 `输入数据 / 说明 / 待确认项`。
5. 输出 `测试执行清单.md` 草稿并确认。
6. 运行 `validate_test_data.py` 校验结构完整性。

这个设计有一个很重要的边界：如果规则强依赖技术方案，但技术方案缺失，Skill 不会强行“脑补正确答案”，而是要求补齐材料，或者把不确定内容显式保留为 `待确认项`。

## 推荐使用方式

如果你的目标是把一份需求真正落成测试资产，推荐按下面的顺序使用：

1. 给 `test-case-generator` 提供需求文档路径。
2. 确认第一阶段生成的 `测试场景地图`。
3. 确认第二阶段生成的 `结构化测试用例`。
4. 如果要继续落测试数据，再补充技术方案与 `reference-pack`。
5. 再调用 `test-data-generator` 生成 `测试执行清单`。

这意味着它不是“一次性吐完整套文档”的黑盒，而是一个带确认点、带校验点的测试设计工作流。

## 真实案例说明

结合当前项目中的真实样例材料，可以看到这套流程是如何工作的。

案例主题为：

`【采购-订单模块】M4X订单管理：查询条件及列表字段调整、新增分配采购员功能`

在这次真实会话里，用户先基于需求文档触发测试用例生成。Skill 没有直接输出最终用例，而是先产出第一阶段 `测试场景地图`，再等待用户确认。用户在对话中补充并确认了多条关键规则，例如：

- `账号` 查询按精确匹配处理。
- `订单状态` 按明确枚举值展开。
- 所有订单状态都允许修改采购员。
- 当前订单采购员不能把订单分配给自己。
- 历史数据修复按一次性脚本验证。

在这些规则被确认后，Skill 才继续生成第二阶段 `结构化测试用例`，随后又执行结构校验。等用户继续推进到测试数据阶段时，Skill 也没有直接产出执行清单，而是先判断当前需求涉及状态映射、历史修复、跨系统同步、权限控制和日志规则，因此必须补齐技术方案与 `reference-pack`，再继续生成 `测试执行清单`。

这次案例最终形成了三份核心输出物：

- `测试场景地图.md`
- `结构化测试用例.md`
- `测试执行清单.md`

这说明 `skill-learn` 的价值不只是“生成一份文档”，而是把测试设计链路拆成了可确认、可追溯、可校验的三个阶段。

## 输出物长什么样

从当前样例可以看到，三个输出物分别承担不同职责：

- `测试场景地图`：回答“这次要测哪些模块、需求点、场景组、补充维度和边界”。
- `结构化测试用例`：回答“每条测试如何执行，前置条件、步骤、预期结果是什么”。
- `测试执行清单`：回答“每条 `TC` 具体依赖哪些表、哪些字段、什么输入数据、哪些地方还待确认”。

这种分层方式比单独维护一份大而全的测试文档更清晰，也更适合和研发、产品、测试共同确认。

## 项目结构

仓库当前的核心结构如下：

```text
skill_learn/
├── .claude-plugin/              # 插件元信息与发布配置
├── skills/
│   ├── test-case-generator/     # 测试用例生成 Skill
│   └── test-data-generator/     # 测试执行清单生成 Skill
├── reference-packs/             # 表结构与样例数据参考包
└── docs/                        # 项目说明与补充文档
```

每个 Skill 都遵循统一组织方式：

- `SKILL.md`：定义触发条件与工作流程。
- `references/`：提供设计规则、覆盖检查清单、质量标准等参考资料。
- `assets/`：提供输出模板与示例。
- `scripts/`：提供结构校验等确定性脚本。

## 安装与使用

插件安装方式如下：

```bash
claude plugins marketplace add "changye01/skill_learn"
claude plugins install "skill-learn"
```

安装完成后，可以直接用自然语言触发，例如：

- “请根据这个需求文档生成测试用例”
- “先帮我梳理测试场景地图”
- “根据已确认测试用例、技术方案和 reference-pack 生成测试执行清单”

## 适用场景

`skill-learn` 特别适合以下场景：

- 需求刚落地，需要先快速完成测试分析与覆盖梳理。
- 需求涉及权限、日志、历史修复、状态映射等复杂规则，单靠人工口述容易遗漏。
- 测试人员希望把“测试设计”与“测试数据准备”分开管理。
- 团队需要一套稳定、可复用、可校验的 Markdown 测试资产产出方式。

## 边界与注意事项

这个项目也有明确边界：

- 它优先处理功能测试设计，不默认覆盖性能、兼容性、安全专项。
- `test-data-generator` 对技术方案依赖较强，缺少技术方案时只能输出低置信度草稿或待确认项。
- “覆盖全面”不等于机械穷举，项目本身也强调避免低价值重复。
- 用户确认是流程的一部分，不能跳过中间确认点直接把草稿当正式结果。

## 一句话总结

`skill-learn` 不是简单的“AI 帮你写测试用例”，而是一套把需求分析、场景确认、结构化用例设计、测试数据补齐和 Markdown 校验串起来的测试设计工作流插件。
