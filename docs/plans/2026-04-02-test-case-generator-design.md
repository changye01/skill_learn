# Test Case Generator Skill 设计文档

日期: 2026-04-02

## 目标

创建一个面向需求文档的测试用例生成 Skill。该 Skill 默认以“覆盖全面”为目标，采用“两阶段生成”流程：先输出场景地图，再展开为结构化测试用例，帮助测试人员在保证覆盖率的前提下控制冗余和颗粒度。

## 设计原则

1. 先场景，后用例。
2. 覆盖全面，但不以数量最大化为目标。
3. 同时考虑显式需求和隐含影响点。
4. 默认补齐权限、日志、数据一致性、持久化、综合流程等维度。
5. 超范围项只提示，不默认混入功能测试主体。

## 各组件职责

| 组件 | 职责 | 加载时机 |
|------|------|----------|
| `SKILL.md` frontmatter | 定义触发条件和适用场景 | 始终在上下文 |
| `SKILL.md` body | 定义引导问题、工作流和输出约束 | 触发后加载 |
| `README.md` | 面向维护者解释规则、原则、输出物和使用方式 | 人工查看 |
| `references/test-design-rules.md` | 约束如何从需求点扩展出测试场景 | 生成场景时读取 |
| `references/coverage-checklist.md` | 约束生成后如何补漏和自检 | 场景地图产出后、用户确认前读取 |
| `assets/scene-template.md` | 第一阶段场景地图输出模板 | 输出场景时读取 |
| `assets/test-case-template.md` | 第二阶段结构化用例模板 | 输出用例时读取 |
| `scripts/validate_cases.py` | 校验输出文件结构与关键字段完整性 | 输出后执行 |

## 适用范围

适用于以下场景：

- 基于需求文档自动生成功能测试场景和测试用例。
- 结合技术文档、接口文档、原型、已有用例补全测试覆盖。
- 对生成结果进行覆盖检查和格式校验。

默认不包含以下内容，除非用户明确要求：

- 性能测试
- 兼容性测试
- 安全测试
- 自动生成测试数据
- 自动生成操作手册

## 输入与输出

### 输入

必选输入：

- 需求文档

可选输入：

- 技术设计文档
- 接口文档
- 页面原型或截图
- 数据库结构说明
- 已有测试用例
- 用户给出的范围约束和优先级

### 输出

第一阶段输出：

- 基于 `assets/scene-template.md` 的 Markdown 场景地图
- 包含场景分组、测试点、补充维度、信息不足待确认项、超范围项

第二阶段输出：

- 基于 `assets/test-case-template.md` 的结构化测试用例
- 包含需求名称、编号、模块、场景组、标题、前置条件、步骤、预期结果、优先级、关联需求点

## 工作流程

```text
用户提供需求材料
→ 确认是否存在补充文档和范围约束
→ 拆解需求点
→ 读取测试设计规则
→ 生成场景地图
→ 读取覆盖检查清单进行补漏
→ 标记信息不足项和超范围项
→ 用户确认场景地图
→ 生成结构化测试用例
→ 运行验证脚本检查完整性
```

## 场景生成策略

每个需求点默认检查以下维度：

- 正常流程
- 边界条件
- 异常输入
- 权限控制
- 状态流转
- 日志记录
- 数据一致性
- 数据持久化
- 历史数据处理
- 端到端综合流程

当需求涉及字段、列表、查询、同步、批处理、修复、日志、角色权限时，应扩展对应的专项场景。

## 去重与颗粒度控制

- 默认追求覆盖全面，但避免机械拆分成大量高度相似的用例。
- 枚举值较多时，优先覆盖关键状态、边界状态和风险状态，而非全部穷举。
- 单条用例不得同时覆盖多个不相干需求点。
- 不允许多个用例仅标题不同、步骤和预期结果实质相同。

## 验证脚本逻辑

`scripts/validate_cases.py` 第一版只校验第二阶段的结构化测试用例文档，重点检查：

- 是否包含需求名称
- 是否包含场景组
- 是否包含关联需求点
- 是否包含测试步骤
- 是否包含预期结果
- 是否包含优先级
- 是否至少存在一组综合场景或端到端场景

对于第一阶段场景地图，先通过模板结构和人工确认保证质量，不纳入第一版脚本校验范围。

- 支持更强的重复用例检测
- 支持把结构化测试用例导出为 Excel 友好的格式
- 与测试数据生成 Skill 串联
- 增加对非功能测试的受控扩展参数
