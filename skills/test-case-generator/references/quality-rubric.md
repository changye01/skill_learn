# 测试用例内容质量评审标准

用于第一版轻量质量校验，目标不是替代人工评审，而是把“什么叫好用例”固定成统一标准，便于 AI 自查和人工复核。

## 适用时机

- 已完成第一阶段场景地图，并做过覆盖检查
- 已生成第二阶段结构化测试用例 Markdown 或 CSV
- 已通过结构校验脚本 `scripts/validate_cases.py`

## 评审目标

评审内容质量时，优先关注以下问题：

- 是否真的覆盖到了需求，而不是只写出格式正确的表格
- 是否可执行、可验证，而不是表述空泛
- 是否存在明显重复、颗粒度过粗或混测问题
- 是否保留了关键业务风险、权限风险和端到端链路

## 六项核心标准

### 1. 可追溯

判断点：

- 每条用例都能追溯到明确需求点、规则、角色、状态或枚举值
- 关键需求点在用例中有对应覆盖，不是只在场景地图里出现
- 用户补充确认过的规则，已经体现在相关用例中

常见问题：

- 用例标题很泛，看不出在验证哪个业务规则
- 需求里写了角色、状态、枚举，但用例未展开
- 用户已确认的规则仍只存在于对话里，没有落到用例

### 2. 可执行

判断点：

- 前置条件具体，执行人能准备数据和进入正确状态
- 测试步骤可操作，不依赖隐含知识
- 步骤顺序清楚，没有跳步

常见问题：

- 只写“准备测试数据”“进入页面”但没有说明关键条件
- 步骤把多个动作揉成一句，执行路径不明确
- 用例依赖前一条用例结果，但没有写明

### 3. 可验证

判断点：

- 预期结果可观察、可断言、可复现
- 结果尽量体现页面、接口、数据库、日志中的至少一个验证点
- 成功与失败的标准清晰，不使用空泛描述

常见问题：

- 预期只写“展示正确”“保存成功”“处理正常”
- 没写具体提示文案、状态变化、字段变化或落库结果
- 只写页面结果，不写关键数据是否持久化

### 4. 不混测

判断点：

- 一条用例尽量只验证一个核心业务点
- 同一条用例里的步骤和预期围绕同一个目标展开
- 不把无关需求、无关校验硬拼到一条里

常见问题：

- 一条用例同时验证查询、编辑、导出、日志
- 标题写“新增成功”，步骤却顺便校验权限和同步
- 为了减少条数，把多个状态值压成一条不可定位的用例

### 5. 不重复

判断点：

- 不存在只改标题、实质相同的重复用例
- 关键角色、关键枚举值可以展开，但每条都应有独立价值
- 去重不以牺牲可执行性和可定位性为代价

常见问题：

- 多条用例除了编号不同，其余描述几乎一样
- 为了“全面”机械枚举大量低价值场景
- 本应拆开的关键角色或关键状态被强行合并

### 6. 有覆盖

判断点：

- 关键需求至少覆盖正常、边界、异常
- 涉及角色时覆盖权限正反向
- 涉及状态时覆盖关键状态流转
- 涉及改动落库时覆盖持久化和一致性
- 保留至少一组综合场景或端到端场景

常见问题：

- 只有正常流程，没有异常和边界
- 只有“有权限用户”，没有无权限反向场景
- 只验证页面提示，不验证接口返回或数据落库
- 缺少关键主流程综合场景

## 轻量评分法

第一版推荐使用三档判断，不做复杂打分：

- `通过`：基本满足该项标准，无明显缺口
- `存疑`：可能覆盖到了，但表达不够清楚，执行或验证成本高
- `不通过`：明显缺失、明显重复、明显不可执行或不可验证

如果一条用例在 `可执行` 或 `可验证` 上为 `不通过`，建议直接改写，不要带着进入回归集。

如果一组用例在 `有覆盖` 上为 `不通过`，优先回到场景地图补漏，再重新展开第二阶段用例。

## 建议评审顺序

1. 先看是否覆盖关键需求点、角色、状态和综合场景
2. 再看单条用例是否可执行、可验证
3. 最后看是否有明显重复和混测

## 评审输出模板

评审时建议输出固定结构，便于 AI 和人工对齐：

```markdown
## 质量评审结论

- 总体结论：通过 / 有条件通过 / 不通过

## 主要问题

- [严重] 哪些需求点未覆盖
- [严重] 哪些用例不可执行或不可验证
- [中等] 哪些用例明显重复或颗粒度过粗

## 建议修正

- 补充哪些场景
- 拆分哪些混测用例
- 细化哪些前置条件或预期结果
```

## 一票否决项

出现以下任一情况时，不建议判为高质量用例集：

- 缺少综合场景或端到端场景
- 明确需求点存在漏测
- 明确权限角色未覆盖到角色级别
- 预期结果大量使用“正确”“成功”“正常”等空泛措辞
- 多条用例仅做表面改写，实质重复

## 与结构校验的边界

`scripts/validate_cases.py` 负责检查：

- 字段是否齐全
- 是否包含综合场景或端到端场景
- Markdown 与 CSV 是否一致

本文件负责检查：

- 用例是否真正映射需求
- 用例是否可执行、可验证
- 用例是否重复、混测或漏关键风险

两者应配合使用，不相互替代。
