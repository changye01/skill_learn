# 测试设计规则

## 1. 需求拆解规则

- 必须先把需求拆成可测试对象：页面、接口、任务、日志、历史数据修复。
- 必须把显式需求点和隐含影响点分开标记。
- 每个显式需求点至少映射到一个场景分组。
- 涉及字段新增或调整时，默认检查展示、查询、回显、落库和日志影响。
- 用户在对话中补充并确认的明确规则，必须单独沉淀到第一阶段 `本次确认规则`，禁止只停留在聊天上下文。
- 第一阶段确认版正式落文件时，必须覆盖目标文件，禁止把旧草稿、旧版本或未确认内容追加到确认版后面。

## 2. 场景扩展规则

- 每个需求点必须检查正常流程。
- 每个需求点默认补齐边界条件和异常场景。
- 涉及角色或入口控制时，必须补权限场景。
- 涉及状态变化时，必须补状态流转场景。
- 涉及系统动作或操作记录时，必须补日志场景。
- 涉及上下游数据同步时，必须补数据一致性场景。
- 涉及修改类功能时，默认补数据持久化场景。
- 涉及跨模块链路时，必须补端到端场景。
- 如果需求明确给出多个允许角色，第二阶段默认每个允许角色至少展开 1 条正向用例，且至少补 1 条无权限反向用例。
- 如果需求明确给出少量关键枚举值（如状态值、类型值、角色值），第二阶段默认按 `1 个值至少 1 条用例` 展开；仅在用户明确要求压缩时才允许合并。

## 3. 综合场景规则

- 必须至少保留一组综合场景或端到端场景。
- 综合场景应串起关键主流程，而不是把多个无关功能硬拼在一起。
- 如果对比材料提示存在典型缺口，默认优先补到综合场景里。

## 4. 数据校验规则

- 页面展示、接口返回、数据库记录应尽量形成一致性校验闭环。
- 历史数据修复类需求，默认补修复前后对比。
- 修改成功后，默认补刷新页面或重新查询后的持久化校验。

## 5. 去重规则

- 禁止机械穷举非关键、数量较大的枚举值，除非用户明确要求。
- 禁止多个用例只改标题、步骤和预期结果实质重复。
- 单条用例禁止覆盖多个不相干需求点。
- 默认优先覆盖关键状态、风险状态和边界状态。
- `去重` 不能成为合并关键权限角色或关键枚举值的理由；对业务关键、小规模、已明确的枚举值，应优先保证可执行性和可定位性。

## 6. 输出格式规则

- 第一阶段场景地图默认使用分组标题 + 列表结构。
- 第二阶段结构化测试用例 Markdown 默认按场景地图中的“场景分组”分段，每组使用表格格式，不改写为纯段落式。

## 7. 超范围控制规则

- 性能、兼容性、安全性默认只提示，不进入功能测试主体。
- 如果材料中出现非功能要求，必须单独标记，不得伪装成功能用例。
- 信息不足时必须显式列出假设或待确认项，禁止直接脑补成确定规则。
