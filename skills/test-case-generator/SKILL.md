# 测试用例生成器

## 工作流程

1. **确定输入材料** - 询问需求文档路径，并确认是否有技术文档、接口文档、原型、数据库说明、已有用例可作为补充材料。
2. **拆解需求点** - 先把需求拆成页面、接口、任务、日志、历史数据修复等可测试对象。
3. **加载测试设计规则** - 读取 `references/test-design-rules.md`，按规则扩展正常、边界、异常、权限、日志、数据一致性、持久化和综合场景。
4. **生成场景地图** - 基于 `assets/scene-template.md` 先输出 Markdown 场景地图，明确“测什么”，并单独保存为一个独立文件。
5. **覆盖检查与补漏** - 读取 `references/coverage-checklist.md`，补充缺失维度，标记信息不足待确认项和超范围项。
6. **确认场景地图** - 在用户确认场景地图前，不得直接展开完整测试用例。
7. **生成结构化测试用例** - 基于确认后的场景地图，使用 `assets/test-case-template.md` 输出结构化测试用例 Markdown，并同步使用 `assets/test-case-template.csv` 输出一个 UTF-8 编码、中文列名的 CSV 文件。第二阶段最终输出模板固定为 5 列：`编号 / 测试功能点 / 前置条件 / 测试步骤 / 预期结果`。
8. **验证完整性** - 运行 `scripts/validate_cases.py` 检查结构化测试用例 Markdown 是否包含关键字段和综合场景，并确认 CSV 与 Markdown 字段一致。

## 默认策略

- 默认目标为`覆盖全面`
- 默认流程为`先场景，后用例`
- 默认补齐权限、日志、数据一致性、数据持久化、综合流程
- 非功能测试默认不纳入功能测试主体，只在超范围项中提示
- 覆盖全面不等于机械穷举，禁止为了“全面”输出大量重复用例
- 第一阶段必须单独输出：`测试场景地图.md`
- 第二阶段默认双输出：`结构化测试用例.md + 表格版用例.csv`
- CSV 默认使用 UTF-8 编码和中文列名，适合直接用 Excel 打开或导入测试平台

## 引导问题

### 第 1 步：确定输入材料

- 请提供需求文档路径。
- 是否还有技术文档、接口文档、页面原型、数据库说明或已有测试用例可一起参考？
- 优先使用 `Markdown (.md)` 作为输入；`txt` 次之，`docx` 和 `pdf` 可用但建议先转成纯文本或 Markdown。
- 如果输入是 `.docx`，优先建议先转成 `.md`，例如：

```bash
pandoc "需求.docx" -t gfm --extract-media=. -o "需求.md"
```

- 转换后应快速检查标题层级、表格、编号和图片引用是否正常，再继续生成测试用例。

### 第 2 步：确定范围与目标

- 本次默认按“覆盖全面”生成，是否有明确不测范围？
- 是否需要先输出场景地图供确认，再展开结构化测试用例？

## 输出要求

### 第一阶段：场景地图

必须包含：

- 功能模块
- 需求点列表
- 场景分组
- 补充维度
- 信息不足待确认项
- 超范围项

第一阶段输出要求：

- 必须单独保存为独立 Markdown 文件，不能只停留在对话中
- 不得被第二阶段输出覆盖
- 推荐文件名：`<需求名称>_测试场景地图.md`

### 第二阶段：结构化测试用例

必须包含：

- 编号
- 测试功能点
- 前置条件
- 测试步骤
- 预期结果

除 Markdown 外，还必须同步生成一个 CSV 文件，列名固定为：

- 编号
- 测试功能点
- 前置条件
- 测试步骤
- 预期结果

CSV 输出要求：

- 使用 UTF-8 编码
- 使用中文列名
- 正确转义逗号、引号和换行
- `测试步骤`、`预期结果` 允许单元格内多行
- `模块`、`场景组`、`关联需求点`、`优先级` 等信息默认保留在第一阶段场景地图中，不再作为第二阶段默认列

## 验证

生成结构化测试用例后，运行：

```bash
python scripts/validate_cases.py <测试用例文件路径>
```

如果提示缺少关键字段或缺少综合场景，根据提示补充。
