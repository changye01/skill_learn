# 测试用例生成器

## 工作流程

1. **确定输入材料** - 询问需求文档路径，并确认是否有技术文档、接口文档、原型、数据库说明、已有用例可作为补充材料。
2. **拆解需求点** - 先把需求拆成页面、接口、任务、日志、历史数据修复等可测试对象。
3. **加载测试设计规则** - 读取 `references/test-design-rules.md`，按规则扩展正常、边界、异常、权限、日志、数据一致性、持久化和综合场景。
4. **生成场景地图** - 基于 `assets/scene-template.md` 先输出 Markdown 场景地图，明确“测什么”。
5. **覆盖检查与补漏** - 读取 `references/coverage-checklist.md`，补充缺失维度，标记信息不足待确认项和超范围项。
6. **确认场景地图** - 在用户确认场景地图前，不得直接展开完整测试用例。
7. **生成结构化测试用例** - 基于确认后的场景地图，使用 `assets/test-case-template.md` 输出结构化测试用例，明确“怎么测”。
8. **验证完整性** - 运行 `scripts/validate_cases.py` 检查结构化测试用例是否包含关键字段和综合场景。

## 默认策略

- 默认目标为`覆盖全面`
- 默认流程为`先场景，后用例`
- 默认补齐权限、日志、数据一致性、数据持久化、综合流程
- 非功能测试默认不纳入功能测试主体，只在超范围项中提示
- 覆盖全面不等于机械穷举，禁止为了“全面”输出大量重复用例

## 引导问题

### 第 1 步：确定输入材料

- 请提供需求文档路径。
- 是否还有技术文档、接口文档、页面原型、数据库说明或已有测试用例可一起参考？

### 第 2 步：确定范围与目标

- 本次默认按“覆盖全面”生成，是否有明确不测范围？
- 是否需要先输出场景地图供确认，再展开结构化测试用例？

## 输出要求

### 第一阶段：场景地图

必须包含：

- 功能模块
- 需求点列表
- 场景分组
- 补充维度
- 信息不足待确认项
- 超范围项

### 第二阶段：结构化测试用例

必须包含：

- 需求名称
- 用例编号
- 模块
- 场景组
- 用例标题
- 关联需求点
- 前置条件
- 测试步骤
- 预期结果
- 优先级

## 资源

- **测试设计规则**: 见 `references/test-design-rules.md`
- **覆盖检查清单**: 见 `references/coverage-checklist.md`
- **场景地图模板**: 见 `assets/scene-template.md`
- **测试用例模板**: 见 `assets/test-case-template.md`

## 验证

生成结构化测试用例后，运行：

```bash
python scripts/validate_cases.py <测试用例文件路径>
```

如果提示缺少关键字段或缺少综合场景，根据提示补充。
