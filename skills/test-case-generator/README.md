# Test Case Generator Skill

一个基于需求文档自动生成测试场景和结构化测试用例的 Skill。

这里的“可执行测试用例”指测试人员可以直接据此理解和执行步骤的基线稿；若还需要把每条 `TC` 对应的数据准备、表记录和执行备注补齐，应继续串联 `test-data-generator` 生成 `测试执行清单.md`。

## 目标

把“需求文档 -> 可执行测试用例”的过程拆成两个清晰阶段：

1. 先生成场景地图，确保覆盖面正确。
2. 再展开结构化测试用例，确保输出可执行。

默认目标是`覆盖全面`，但不以数量最大化为目标。

## 适用范围

适用于以下场景：

- 根据需求文档生成功能测试用例
- 结合技术文档、接口文档、原型、已有用例补齐覆盖
- 先做场景评审，再输出结构化测试用例

默认不包含以下内容，除非用户明确要求：

- 性能测试
- 兼容性测试
- 安全测试
- 测试数据生成
- 操作手册生成

## 设计原则

- 先场景，后用例
- 覆盖全面，但不以数量最大化为目标
- 同时考虑显式需求和隐含影响点
- 默认补齐权限、日志、数据一致性、数据持久化、综合流程
- 超范围项只提示，不默认混入功能测试主体
- 禁止把大量仅有枚举值差异的重复内容机械拆成多条用例

## 两阶段生成流程

### 第一阶段：场景地图

先在对话中输出 Markdown 场景地图草稿，重点回答“测什么”。

内容包括：

- 功能模块
- 需求点列表
- 场景分组
- 补充维度
- 信息不足待确认项
- 超范围项

其中 `超范围项` 用来明确记录“已经识别到、但当前版本不纳入功能测试主体”的内容，常见如性能、兼容性、安全性、大批量压测、长稳测试等。保留这一栏可以避免 AI 直接把这些点混进主用例集，也能让评审者分清哪些内容是有意排除而不是遗漏。

用户需要先整体确认这份场景地图草稿，包括：

- 场景分组是否合理
- 是否有漏场景或冗余场景
- 信息不足待确认项是否准确
- 超范围项是否合理

确认后，再将第一阶段结果单独落成文件，推荐命名为：

- `<需求名称>_测试场景地图.md`

这样可以保留第二阶段展开前的确认版评审依据，避免把未确认草稿直接落盘，也避免后续只剩结构化用例而丢失场景分组、待确认项和超范围项。

默认落地规则：

- 用户提供目录时，默认保存到该目录
- 用户提供需求文档文件时，默认保存到需求文档同目录
- 用户提供明确输出文件路径时，按指定路径保存

另外，用户在对话里补充并确认过的明确规则，建议单独沉淀为 `本次确认规则` 小节，例如：

- 查询是精确匹配还是模糊匹配
- 查询是否大小写敏感
- 权限不足时是隐藏入口还是提示无权限
- 弹窗是否支持搜索、分页
- 某些状态值、角色值、类型值是否已经明确

这样可以避免这些规则只停留在聊天上下文里，导致第二阶段遗漏或误改。

### 第二阶段：结构化测试用例

在用户确认场景地图后，再展开结构化测试用例，重点回答“怎么测”。

内容包括：

- 编号
- 测试功能点
- 前置条件
- 测试步骤
- 预期结果

这 5 列就是第二阶段的最终输出模板，贴近日常测试执行习惯。`模块`、`场景组`、`关联需求点`、`优先级` 等信息默认保留在第一阶段场景地图里，不再作为第二阶段默认列。

第二阶段默认双输出：

- `结构化测试用例.md`
- `表格版用例.csv`

推荐文件名：

- `<需求名称>_结构化测试用例.md`
- `<需求名称>_表格版用例.csv`

第二阶段闭环要求：

- 用户确认第二阶段结果后，必须先保存这两份文件，不能只输出到对话中
- 保存时应覆盖旧版本，不能把历史草稿追加到确认版文件里
- 保存后必须立即运行校验脚本
- 校验通过前，不能视为确认版，也不应直接提示下游 `test-data-generator`

### Markdown 形式建议

第二阶段 `Markdown` 默认建议继续使用“按场景分组 + 组内表格”的格式，不建议改成纯段落式。

原因：

- 更适合保留第一阶段场景地图中的分组语义
- 更适合和 `CSV` 保持一一对应
- 更适合人工评审、筛选、对比和补漏
- 更适合后续导入 Excel 或测试平台
- 更容易发现缺列、漏列、空字段、重复编号等结构问题

因此建议固定为：

- 第一阶段场景地图：分组标题 + 列表
- 第二阶段结构化测试用例：按场景组分段，每组使用 5 列 Markdown 表格

具体建议：

- Markdown 中的分组名称与顺序继承第一阶段 `场景分组`
- CSV 保持平铺结构，不额外增加“场景组”列
- CSV 中的编号顺序应与 Markdown 按场景组展开后的顺序一致

除非用户明确要求“段落式用例说明”，否则不建议把第二阶段默认格式改成段落叙述。

## 输出物说明

- `assets/scene-template.md`
  第一阶段场景地图模板
- `assets/test-case-template.md`
  第二阶段结构化测试用例 Markdown 模板
- `assets/test-case-template.csv`
  第二阶段表格版测试用例 CSV 模板
- `assets/test-case-example.md`

- `assets/test-case-example.csv`

## 使用方式

### 推荐输入格式

建议按以下优先级提供需求材料：

1. `Markdown (.md)`
2. `UTF-8 纯文本 (.txt)`
3. `Word (.docx)`，建议先转 `Markdown`
4. `PDF`

说明：

- `Markdown` 最适合 AI 直接读取结构，标题、列表、表格更稳定。
- `docx` 可用，但复杂排版、表格和批注可能影响解析质量。
- `pdf` 可直接使用，但容易出现换行错乱、表格断裂、页眉页脚混入正文等问题。

### 飞书文档转 Markdown

如果原始需求在飞书文档中，推荐使用 Chrome 浏览器里的 `Cloud Document Converter` 将飞书文档转换成 `Markdown`，再交给本 Skill。

推荐步骤：

1. 在 Chrome 中打开飞书需求文档
2. 使用 `Cloud Document Converter` 将当前飞书文档转换为 `Markdown`
3. 将转换结果保存为本地 `.md` 文件，例如 `需求提取_26668.md`
4. 把这份 `Markdown` 文件作为输入交给本 Skill

说明：

- 这种方式更贴合当前“飞书文档 -> Markdown -> Skill”的实际使用路径。
- 转换完成后，仍建议人工检查标题层级、表格、编号、代码块和图片链接是否正常。
- 如果飞书原文里有复杂表格、嵌套列表或大量截图，建议在交给 Skill 前先做一次轻量清洗。

建议输入顺序：

1. 需求文档
2. 技术文档、接口文档、原型、数据库说明等补充材料
3. 生成目标和范围约束

建议执行顺序：

1. 先在对话中生成场景地图草稿
2. 用户确认整份场景地图
3. 再保存确认版场景地图文件
4. 生成结构化测试用例 Markdown
5. 用户确认第二阶段结果
6. 同步保存 `结构化测试用例.md` 和中文列名的 `CSV`
7. 运行校验脚本检查 Markdown 完整性，并确认配套 CSV 存在且字段一致
8. 参考 `references/quality-rubric.md` 做一轮内容质量自查

补充建议：

- 正式保存确认版场景地图时，应覆盖目标文件，不能把旧草稿追加进去
- 如果用户只给目录，默认把两个阶段的输出都落到该目录；如果用户给的是需求文档文件，默认落到需求文档同目录
- 若需求里明确列出多个可操作角色，第二阶段应至少为每个允许角色各给 1 条正向用例，并补 1 条无权限反向用例
- 若需求里明确列出少量关键枚举值（如状态值），默认按 `1 个值至少 1 条用例` 展开；只有用户明确要求压缩时才合并

### CSV 输出约定

第二阶段 CSV 默认使用中文列名：

- `编号`
- `测试功能点`
- `前置条件`
- `测试步骤`
- `预期结果`

生成 CSV 时建议遵循：

- 使用 UTF-8 编码
- 正确转义逗号、引号和换行
- `测试步骤`、`预期结果` 允许单元格内多行
- 适合作为 Excel 或测试平台导入的表格版输出

## 验证方式

对结构化测试用例 Markdown 文档运行：

```bash
python scripts/validate_cases.py <测试用例文件路径> [CSV 文件路径]
```

脚本会检查：

- 是否包含关键字段
- 是否包含综合场景或端到端场景
- 是否存在配套 CSV 文件
- CSV 列头、编号集合和顺序是否与 Markdown 一致

如果不显式传入 CSV 路径，脚本默认会在同目录下先检查同名 `.csv`，并兼容检查 `<需求名称>_表格版用例.csv`。

完成结构校验后，建议再参考 `references/quality-rubric.md` 做一轮轻量内容质量评审，重点检查：

- 是否真正映射到需求点、角色、状态和关键枚举值
- 前置条件、步骤、预期结果是否足够可执行、可验证
- 是否存在明显重复、混测或颗粒度过粗的问题
- 是否保留关键异常、权限反向和综合场景

第一阶段场景地图默认依赖模板约束和人工确认，不纳入第一版脚本校验范围。第一版推荐采用“结构校验 + 内容质量评审”组合，而不是只校验 Markdown 格式。

如果下一步目标是“执行这些测试用例需要准备哪些数据”，建议在完成上述验证后继续搭配 `test-data-generator`：

1. 先确认 `结构化测试用例.md` / `表格版用例.csv`
2. 再提供技术方案 / 接口说明
3. 再提供对应的 `reference-pack` 路径（通常位于 `reference-packs/...`，包含 `tables/*.sql` + `table_samples/*.csv`）
4. 由 `test-data-generator` 保留 `结构化测试用例.md` 作为基线稿，并进一步输出按场景组分组、组内按 `TC` 展开的 `测试执行清单.md`

其中技术方案不是可有可无的补充材料；如果存在派生字段、状态映射、历史修复、权限规则或日志映射，应优先补齐技术方案后再进入 `test-data-generator`。
