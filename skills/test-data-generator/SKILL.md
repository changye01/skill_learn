# 测试数据生成器

## 工作流程

1. **确认输入材料** - 询问测试用例路径，并确认是否已提供 `reference-pack`（`tables/*.sql` + `table_samples/*.csv`）；若还有技术方案、接口说明、状态映射说明，可一并作为补充参考。
2. **识别是否需要补充技术方案** - 如果存在派生字段、状态映射、历史修复、跨系统同步、接口返回字段不是数据库原字段等情况，应主动提示“建议补充技术方案作为参考”。
3. **拆解测试数据需求** - 以测试用例为主，先梳理每条用例需要的是已有数据、新建数据、异常数据、缺失数据、边界数据还是历史修复前后数据。
4. **读取 reference-pack** - 读取 `reference-packs/tables/*.sql` 了解表结构，读取 `reference-packs/table_samples/*.csv` 了解字段形态与样例分布。
5. **输出测试数据设计草稿** - 基于 `assets/test-data-template.md` 在对话中先输出 Markdown 测试数据设计草稿，明确“为哪条用例准备什么数据”。
6. **覆盖检查与补漏** - 读取 `references/data-coverage-checklist.md` 检查是否覆盖正常、边界、异常、缺失、历史修复、派生字段、状态映射和多表关联等关键维度。
7. **沉淀待确认项** - 对无法从测试用例和 reference-pack 可靠得出的规则，单独输出为 `待确认项`；禁止直接脑补成确定规则。
8. **保存确认版测试数据设计** - 用户确认后，再将结果保存为独立 Markdown 文件；保存时必须覆盖目标文件，不得把旧草稿追加到确认版文件中。
9. **验证完整性** - 运行 `scripts/validate_test_data.py` 检查测试数据设计 Markdown 是否包含关键字段和至少一条数据设计记录。

## 默认策略

- 默认输入优先级为：`测试用例 > reference-pack > 技术方案`
- 默认把技术方案视为`补充参考`，不是必需主输入
- 默认输出测试数据设计，而不是直接生成 SQL/JSON/fixture
- 默认保留 `待确认项`，不把推断内容伪装成确定规则
- 默认优先复用 `reference-pack/table_samples/*.csv` 中已存在的数据形态
- 默认补齐正常、边界、异常、缺失、历史修复、多表关联、派生字段相关数据
- 覆盖完整不等于机械穷举，禁止为“全面”输出大量低价值重复数据设计

## 引导问题

### 第 1 步：确定输入材料

- 请提供测试用例路径。
- 请提供对应的 `reference-pack` 路径。
- 是否还有技术方案、接口说明、状态映射说明或修复 SQL 可一起参考？
- 默认优先使用 `Markdown (.md)` 作为测试用例输入；`CSV` 可作为补充或备选。

### 第 2 步：识别补充材料需求

- 当前需求是否包含派生字段、状态映射、历史修复、跨系统同步等规则？
- 如果有但尚未提供技术方案，是否需要先补充相关文档再继续？

## 输出要求

### 测试数据设计

必须包含：

- 编号
- 测试用例编号
- 数据目标
- 涉及表
- 关键字段
- 建议取值/样例来源
- 数据操作
- 关联关系
- 备注/待确认项

输出要求：

- 必须先在对话中给出测试数据设计草稿，供用户审阅
- 用户未确认前，不得保存正式文件
- 用户确认后，才保存为独立 Markdown 文件
- 对无法确认的规则必须保留在 `备注/待确认项`
- 若技术方案缺失但规则依赖强，应显式提示“建议补充技术方案作为参考”
- 推荐文件名：`<需求名称>_测试数据设计.md`

## 资源

- **测试数据设计规则**: 见 `references/data-design-rules.md`
- **覆盖检查清单**: 见 `references/data-coverage-checklist.md`
- **测试数据模板**: 见 `assets/test-data-template.md`

## 验证

生成测试数据设计后，运行：

```bash
python scripts/validate_test_data.py <测试数据设计文件路径>
```

如果提示缺少关键字段、缺少数据设计记录，根据提示补充；如果材料不足以支撑确定性设计，应补充到 `待确认项`，而不是直接编造。
