# 测试执行清单生成器

## 工作流程

1. **确认输入材料** - 询问已确认的 `结构化测试用例.md` 路径，并确认是否已提供技术方案与 `reference-pack`（`tables/*.sql` + `table_samples/*.csv`）；技术方案用于定义规则，`reference-pack` 用于确认落表字段与样例形态。
2. **判断技术方案是否为必需输入** - 如果存在派生字段、状态映射、历史修复、跨系统同步、接口返回字段不是数据库原字段、权限判断或日志展示规则等情况，应明确提示“缺少技术方案，无法给出高置信度测试数据设计”。
3. **继承基线用例** - 逐条读取已确认测试用例中的 `TC`、测试功能点、前置条件、测试步骤、预期结果，把它们作为执行稿基线，不在这一阶段随意改写测试目标。
4. **先读技术方案，再按 reference-pack 缩圈** - 先从技术方案中确认字段映射、状态语义、权限规则、日志来源和历史修复逻辑；如果 `reference-pack` 带有 `table_index.csv`，必须先读取索引，根据业务词、表名、表注释和关键字段缩圈出候选表，再读取对应 `tables/*.sql` 与 `table_samples/*.csv`；禁止默认整包通读全量表文件。
5. **按 TC 拆解测试数据需求** - 以单条用例为单位，判断该 `TC` 需要的是已有数据、新建数据、异常数据、缺失数据、边界数据还是历史修复前后数据。
6. **输出 Markdown 测试执行清单草稿** - 基于 `assets/test-data-template.md` 在对话中先输出 `测试执行清单.md` 草稿，默认按场景地图中的“场景分组”组织；每个场景组下先给阅读摘要，再按 `TC` 全量展开，其中每条 `TC` 先放 5 列用例表，再补充 `输入数据 / 说明 / 待确认项`。
7. **覆盖检查与补漏** - 读取 `references/data-coverage-checklist.md`，检查每条 `TC` 是否都具备 5 列用例表、输入数据、说明和待确认项，并补齐正常、边界、异常、缺失、历史修复、派生字段、状态映射和多表关联等关键维度。
8. **沉淀待确认项** - 对无法从测试用例、技术方案和 reference-pack 可靠得出的规则，单独输出到对应 `TC` 的 `待确认项`；禁止直接脑补成确定规则。
9. **按批保存候选版 Markdown** - 用户确认草稿方向后，再将结果保存为独立 Markdown 文件。默认按“文档头部 + 场景组”分批保存到同一目标文件：第一批必须写入文档标题、`输入材料`、`阅读导航` 和第一个场景组，后续批次按完整场景组追加；如果单个场景组仍然过大，再按完整 `TC` 批次追加。禁止把旧草稿追加到候选版文件中，也禁止把半个场景组、半个 `TC`、半张表或半个小节拆到不同批次。
10. **验证 Markdown 完整性** - 运行 `scripts/validate_test_data.py` 检查 `测试执行清单.md` 是否符合规范。
11. **校验失败则回改 Markdown 并重新校验** - 如果校验未通过，必须先回改同一份 Markdown，再重新运行校验；只有校验通过后，这份 Markdown 才能视为确认版。

## 默认策略

- 默认输入优先级为：`结构化测试用例 + 技术方案 > reference-pack`
- 默认技术方案是核心输入之一，不再视为可有可无的补充材料
- 默认正式输出就是 `测试执行清单.md`
- 默认先给 Markdown 草稿，保存候选版并完成校验，通过后才算确认版
- 默认保留 `待确认项`，不把推断内容伪装成确定规则
- 默认优先复用 `reference-pack` 路径下 `table_samples/*.csv` 中已存在的数据形态
- 如果 `reference-pack` 包含大量表，默认只把它当作候选库，必须先用 `table_index.csv` 或明确表名缩圈
- 默认单次只展开少量候选表；通常先控制在 `3-10` 张，证据不足时再逐步扩表
- 默认输出时要说明“为什么选这些表”，而不是只给结论不交代依据
- 默认不再以“基础记录池”组织正文；同一条数据命中多个 `TC` 时，允许在多个 `TC` 下重复展示
- 默认测试执行清单继承场景地图中的场景分组名称与顺序
- 默认在 `输入材料` 后增加 `阅读导航`，便于快速定位场景组
- 默认每个场景组先给“本组目标 + 摘要表”，再展开 `TC`
- 默认 `输入数据` 小节按表分块展示，不跨表混排
- 默认单条 `TC` 在标题下先给一张 5 列用例表（`编号 / 测试功能点 / 前置条件 / 测试步骤 / 预期结果`）
- 默认每张表使用 `**表：表名**` 作为块标签，不再额外增加表级标题
- 默认每张表增加简短“用途”标注，如“主断言表 / 关联表 / 日志表”
- 默认每张表只保留“关键断言字段 + 关联字段 + 必要上下文字段”的最小充分记录
- 默认保存候选版时优先按场景组分批写入同一文件；只有单个场景组仍然过大时，才退化为按 `TC` 分批写入
- 默认第一批保存必须包含文档头部、`输入材料`、`阅读导航` 和第一个场景组，不能只写正文片段
- 默认全部批次保存完成后再统一运行校验脚本，不在中间批次重复校验
- 如果关键规则依赖技术方案而当前缺失，只能输出带较多 `待确认项` 的草稿，必要时应先停下来要求补齐材料
- 覆盖完整不等于机械穷举，禁止为了“全面”输出大量低价值重复数据设计

## 引导问题

### 第 1 步：确定输入材料

- 请提供已确认的 `结构化测试用例.md` 路径。
- 请提供对应技术方案路径。
- 请提供对应的 `reference-pack` 路径。
- 如果该 `reference-pack` 是全库级或表很多，是否已提供 `table_index.csv` 或候选表名单？
- 是否还有接口说明、状态映射说明、修复 SQL 或其它补充材料可一起参考？
- 默认优先使用 `Markdown (.md)` 作为测试用例输入。

### 第 2 步：识别补充材料需求

- 当前需求是否包含派生字段、状态映射、历史修复、跨系统同步等规则？
- 如果有但尚未提供技术方案，是否先暂停测试执行清单补齐，待补齐技术方案后再继续？

## 输出要求

### Markdown 测试执行清单

必须包含：

- 文档标题包含 `测试执行清单`
- 至少一条 `TC-xxx 测试功能点`
- 推荐结构为“场景组 > TC > 小节”
- 推荐在 `输入材料` 后增加 `阅读导航`
- 推荐每个场景组先给“本组目标 + 摘要表”
- 每条 `TC` 下有一张 5 列用例表：`编号 / 测试功能点 / 前置条件 / 测试步骤 / 预期结果`
- `输入数据`
- `说明`
- `待确认项`

输出要求：

- 必须先在对话中给出 Markdown 草稿，供用户审阅
- 用户未确认前，不得保存正式文件
- 用户确认草稿方向后，可保存候选版 Markdown
- 只有候选版通过校验后，才可视为正式确认版
- 保存候选版时，默认按“第一批文档头部 + 第一个场景组、后续批次逐个场景组”的顺序分批写入同一目标文件
- 如果单个场景组内容仍然过大，可在该场景组内部再按完整 `TC` 分批写入，但不得把一个 `TC` 拆散到多个批次
- 每一批都必须保持 Markdown 结构完整，禁止把半张表、半个小节或半个场景组写到不同批次
- 全部批次写完后，才运行 `validate_test_data.py` 做最终校验
- 默认按场景地图中的“场景分组”组织正文；场景组名称与顺序应与基线稿保持一致
- 推荐在正文前增加 `阅读导航`，便于测试人员快速跳转
- 每个场景组建议先用 1 句话写清“本组目标”，再给一张摘要表（如 `编号 / 测试功能点 / 数据重点`）
- `TC` 标题用于定位编号与功能点，正文中再用 5 列表格承载完整基线信息
- 单条 `TC` 默认按 `5 列用例表 -> 输入数据 -> 说明 -> 待确认项` 排列
- `输入数据` 必须按表分块展示，使用 `**表：表名**` + Markdown 表格承载示例记录
- 每张表建议增加简短用途标注，帮助区分“主断言表 / 关联表 / 日志表 / 上下文表”
- 如果 `reference-pack` 来自全库导出，必须先写明本次使用的候选表清单与选表依据，禁止在未缩圈前整包读取
- 当候选表超过 `10` 张时，应先停下来收敛范围，必要时要求用户补充更明确的业务范围、字段名或表名单
- 对无法确认的规则必须保留在对应 `TC` 的 `待确认项`
- 若技术方案缺失但规则依赖强，应显式提示“缺少技术方案，当前只能输出低置信度草稿或待确认项”
- 推荐文件名：`<需求名称>_测试执行清单.md`

## 验证

生成 Markdown 后，运行：

```bash
python scripts/validate_test_data.py <Markdown文件路径>
```

脚本会检查新版 `测试执行清单.md` 的结构是否完整：

- 文档标题是否包含 `测试执行清单`
- 是否至少包含一条 `TC-xxx`
- 是否支持“场景组 > TC > 小节”的嵌套结构
- 每条 `TC` 是否都包含 5 列用例表，以及 `输入数据 / 说明 / 待确认项`
- `输入数据` 是否按 `**表：表名**` 或兼容旧版表级标题分块展示

如果提示缺少必要小节、缺少 `TC-xxx` 或 `输入数据` 未按表分块，必须先回改 Markdown，再重新运行校验，直到通过为止；如果关键规则缺少技术方案支撑，应优先要求补齐方案，至少也要把不确定内容补充到 `待确认项`，而不是直接编造。
