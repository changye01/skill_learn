# 测试数据设计规则

本规则用于把“已确认结构化测试用例”转换成“可直接执行的测试执行清单”。

## 核心原则

### 1. 先用例，后数据

- 先看单条 `TC` 要验证什么，再决定准备什么数据。
- 不要先围绕数据库字段机械列数据，再反推测试目标。

### 2. 基线稿与执行稿分层

- `结构化测试用例.md`：定义 `TC`、前置条件、测试步骤、预期结果。
- `测试执行清单.md`：继承上述内容，并补充测试数据、说明、待确认项。
- `test-data-generator` 不负责重写测试目标，只负责把执行上下文补齐。

### 3. 按 TC 展开，不做正文总表

- 最终阅读入口是单条 `TC`，不是顶部汇总表。
- 同一条数据如果命中多个用例，可在多个 `TC` 下重复展示。
- 禁止为了“去重”牺牲按 `TC` 阅读的顺手程度。

### 4. 按表分块，保留最小充分记录

- `测试数据` 必须按表分块展示，不跨表混排。
- 每张表只保留“关键断言字段 + 关联字段 + 必要上下文字段”。
- 不重复整理完整字段字典，不扩写成通用数据库文档。

### 5. 可确认与待确认分开

- 能从现有材料确认的内容直接写进对应 `TC`。
- 无法确认的规则必须写入对应 `TC` 的 `待确认项`。
- 禁止用经验臆测接口映射、状态语义或修复逻辑。

## 数据目标分类

常见测试数据目标：

- 正常数据：满足主流程执行条件
- 边界数据：命中长度、数量、状态边界
- 异常数据：触发报错、拦截、兜底逻辑
- 缺失数据：字段为空、关联丢失、历史未修复
- 冲突数据：重复、脏数据、状态不一致
- 派生数据：需要经映射、计算、转换后才生效

## 设计方法

### 第 1 步：按测试用例拆数据需求

每条测试用例至少回答：

- 需要已有数据还是新造数据
- 需要命中哪个筛选条件或业务状态
- 是否依赖多表关联
- 是否依赖派生字段或映射规则
- 是否需要历史未修复数据或修复后数据

### 第 2 步：按 reference-pack 确认落点

- 用 `tables/*.sql` 确认字段是否存在、字段属于哪张表。
- 用 `table_samples/*.csv` 确认取值形态、空值形态、示例组合。
- 优先复用样例数据的结构模式，减少脱离真实数据形态的设计。

### 第 3 步：按 TC 组织表记录

每条测试用例的 `测试数据` 至少应体现：

- 涉及哪些表
- 每张表中用于断言和关联的关键字段
- 哪些字段只是提供执行上下文
- 需要人工准备、补造、修改还是保留缺失态

### 第 4 步：补齐说明和待确认项

每条测试用例至少应说明：

- 关键关联关系或执行备注
- 与基线稿相比需要额外注意的上下文
- 哪些内容仍待确认

## 技术方案何时必须补充

出现以下任一情况时，应主动提示补充技术方案：

- 接口返回字段不是数据库原字段
- 字段需要映射成标签、文案或枚举说明
- 历史修复逻辑会影响测试判断
- 同一业务字段来自多表拼装
- 涉及跨系统同步、补偿或异步任务

## 输出要求

主设计稿建议固定使用 Markdown，推荐结构如下：

- `# <需求名称> 测试执行清单`
- `## 输入材料`
- `## TC-xxx 测试功能点`

其中每条测试用例固定包含：

- `前置条件`
- `测试步骤`
- `预期结果`
- `测试数据`
- `说明`
- `待确认项`

## 常见误区

- 只有表名，没有关键字段
- 只有样例来源，没有数据目标
- 仍按“基础记录池 + 引用”组织正文，导致执行时来回跳读
- 把“需要确认的业务规则”写成确定结论
- 明知依赖映射规则，却不提示补充技术方案
- 用大量重复记录堆覆盖，看起来很多，实际信息量很低
