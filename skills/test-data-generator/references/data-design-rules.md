# 测试数据设计规则

本规则用于把“已确认测试用例”转换成“可执行的测试数据清单”。

## 核心原则

### 1. 先用例，后数据

- 先看测试用例要验证什么，再决定准备什么数据。
- 不要先围绕数据库字段机械列数据，再反推用例。

### 2. 主输入与补充输入分层

- 测试用例：决定数据目标和覆盖范围。
- `reference-pack`：决定表结构、字段名称、样例形态。
- 技术方案：补充派生规则、状态映射、历史修复、跨系统链路。

### 3. 先抽公共池，再写用例增量

- 对多条用例重复使用的订单、明细、产品、角色、日志等记录，优先沉淀到 `基础记录池`。
- 每条测试用例优先写“引用基础记录”，只对差异字段、特殊状态、修复前后态做增量补充。
- 禁止在几十条用例下重复粘贴结构完全相同的表记录。

### 4. 只设计当前需要的数据

- 只围绕当前测试用例涉及的表、字段和关系输出。
- 不重复整理完整字段字典，不扩写成通用数据库文档。

### 5. 可确认与待确认分开

- 能从现有材料确认的内容直接写进清单。
- 无法确认的规则必须写入 `待确认项`。
- 禁止用经验臆测接口映射、状态语义或修复逻辑。

## 数据目标分类

常见测试数据目标：

- 正常数据：满足主流程执行条件
- 边界数据：命中长度、数量、状态边界
- 异常数据：触发报错、拦截、兜底逻辑
- 缺失数据：字段为空、关联丢失、历史未修复
- 冲突数据：重复、脏数据、状态不一致
- 派生数据：需要经映射、计算、转换后才生效

## 设计方法

### 第 1 步：按测试用例拆数据需求

每条测试用例至少回答：

- 需要已有数据还是新造数据
- 需要命中哪个筛选条件或业务状态
- 是否依赖多表关联
- 是否依赖派生字段或映射规则
- 是否需要历史未修复数据或修复后数据

### 第 2 步：按 reference-pack 确认落点

- 用 `tables/*.sql` 确认字段是否存在、字段属于哪张表。
- 用 `table_samples/*.csv` 确认取值形态、空值形态、示例组合。
- 优先复用样例数据的结构模式，减少脱离真实数据形态的设计。

### 第 3 步：先抽基础记录池

基础记录池至少应体现：

- 哪张表里有哪些可复用记录
- 这些记录的主键/业务键是什么
- 哪些字段是后续测试人员最需要看的关键值
- 修复前/修复后、角色池、日志池等是否需要单独成组

### 第 4 步：按用例写引用和增量

每条测试用例至少应说明：

- 引用哪些基础记录
- 是否有额外补充或覆盖字段
- 需要验证什么结果
- 哪些内容仍待确认

## 技术方案何时必须补充

出现以下任一情况时，应主动提示补充技术方案：

- 接口返回字段不是数据库原字段
- 字段需要映射成标签、文案或枚举说明
- 历史修复逻辑会影响测试判断
- 同一业务字段来自多表拼装
- 涉及跨系统同步、补偿或异步任务

## 输出要求

主设计稿建议固定使用 Markdown，推荐结构如下：

- `基础记录池`
- `测试用例清单`
- `待确认项`

其中每条测试用例建议包含：

- `引用基础记录`
- `验证点`
- 必要时补一段“补充说明”或“修改后预期”

如用户明确需要，在确认版 Markdown 通过校验后，可再派生一份测试人员使用的 CSV 测试数据清单。

## 常见误区

- 只有表名，没有关键字段
- 只有样例来源，没有数据目标
- 没有抽公共记录池，导致整份文档大量重复
- 把“需要确认的业务规则”写成确定结论
- 明知依赖映射规则，却不提示补充技术方案
- 用大量重复记录堆覆盖，看起来很多，实际信息量很低
