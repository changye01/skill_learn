# 测试执行清单 Skill

一个与 `test-case-generator` 串联使用的 Skill，用于把“已确认测试用例”进一步转成“可直接执行的测试执行清单”。

## 目标

把“结构化测试用例 -> 测试执行清单”的过程单独拆出来，明确回答：

1. 每条 `TC` 执行时到底要准备哪些数据。
2. 这些数据落在哪些表、依赖哪些关联、验证哪些断言。
3. 哪些规则可以确定，哪些必须依赖技术方案才能确认。

默认目标是 `执行信息完整 + 测试数据可直接落到 TC 上下文中阅读`，而不是额外维护一份抽象的数据池总表，也不是直接产出可执行 SQL。

## 与 `test-case-generator` 的关系

推荐使用“基线稿 + 执行稿”模式：

1. `test-case-generator` 先输出并确认 `结构化测试用例.md`，负责定义 `TC`、前置条件、测试步骤、预期结果。
2. `test-data-generator` 再读取已确认用例、技术方案和 `reference-pack`，补充测试数据、说明、待确认项，输出 `测试执行清单.md`。

这样可以把“测什么 / 怎么测”和“拿什么数据执行”分开管理，同时保证最终执行稿按 `TC` 阅读顺手。

## 推荐输出

正式输出统一为 `Markdown` 版 `测试执行清单.md`，推荐结构如下：

1. `# <需求名称> 测试执行清单`
2. `## 输入材料`
3. `## 阅读导航`
4. `## 场景组X：<场景组名称>`
5. `### TC-xxx 测试功能点`

其中每条 `TC` 固定展开：

- `前置条件`
- `输入数据`
- `测试步骤`
- `预期结果`
- `说明`
- `待确认项`

约束说明：

- 不保留顶部总表
- 不再以“基础记录池”作为主阅读结构
- 默认继承场景地图中的场景分组名称与顺序
- 推荐在 `输入材料` 后增加 `阅读导航`，用于快速跳转场景组
- 推荐每个场景组先写 `本组目标`，再给一张轻量摘要表（如 `编号 / 测试功能点 / 数据重点`）
- 同一条数据命中多个 `TC` 时，允许在多个 `TC` 下重复展示
- `输入数据` 必须按表分块展示，不能跨表混排
- 推荐单条 `TC` 先看 `输入数据` 再执行步骤，便于执行前快速准备数据
- 推荐给每张表增加简短用途标注，如“主断言表 / 关联表 / 日志表”
- 每张表只保留“关键断言字段 + 关联字段 + 必要上下文字段”的最小充分记录

## 工作流程

1. 确认已确认的 `结构化测试用例.md`、技术方案和 `reference-pack` 路径。
2. 判断是否存在派生字段、状态映射、历史修复、跨系统链路、权限或日志规则。
3. 若有上述复杂规则但缺少技术方案，应明确提示当前无法给出高置信度执行数据设计。
4. 逐条继承基线用例中的 `TC / 前置条件 / 步骤 / 预期`。
5. 先读技术方案，再读 `reference-pack` 路径（通常位于 `reference-packs/...`）下的 `tables/*.sql` 和 `table_samples/*.csv`，定位每条用例所需的表、字段和值形态。
6. 输出 Markdown `测试执行清单` 草稿，默认按场景组组织正文，先给 `阅读导航` 与场景组摘要，再在组内展开 `TC`。
7. 根据覆盖清单补漏，并把无法确认的信息留在对应 `TC` 的 `待确认项`。
8. 用户确认草稿方向后保存候选版 Markdown。
9. 运行校验脚本做 Markdown 结构校验。
10. 如校验失败，回改 Markdown 并重新校验。
11. 校验通过后，再将其视为确认版。

## 建议输入

### 主输入

1. 已确认的 `结构化测试用例.md`
2. 技术方案
3. `reference-pack` 路径（通常位于 `reference-packs/...`）
   - `tables/*.sql`
   - `table_samples/*.csv`

### 补充输入

- 接口说明
- 状态/枚举映射说明
- 历史修复 SQL 或迁移说明

## 技术方案的重要性

对 `test-data-generator` 来说，技术方案不是普通补充材料，而是核心输入之一。

尤其在以下场景，没有技术方案通常无法给出高置信度的测试执行清单：

- 页面展示字段不是数据库原字段
- 状态值、标签、枚举需要映射
- 历史修复会影响修复前后判断
- 权限规则决定谁能操作、谁不能操作
- 日志展示与日志落表不一致
- 同一业务字段来自多表拼装或跨系统同步

此时 `reference-pack` 只能帮助确认字段落点和样例形态，不能替代技术方案本身。

## `reference-pack` 与 `table_samples` 的作用

`test-data-generator` 会参考 `reference-pack/table_samples/*.csv`，但使用方式不是“整张样例直接照抄进执行清单”，而是：

- 参考样例中的字段形态、取值风格、空值形态和常见字段组合
- 优先复用更接近真实环境的数据模式，避免凭空编造很假的输入数据
- 结合 `tables/*.sql` 确认字段落点后，只保留当前 `TC` 真正需要的最小充分记录

不建议这样使用：

- 把整张 sample 原样搬进 `测试执行清单.md`
- 看到样例里有什么值就机械照抄，不区分当前 `TC` 是否真的需要
- 用 `table_samples` 替代技术方案去判断状态映射、权限规则、日志规则或历史修复逻辑

一句话说，`table_samples` 用来让输入数据“更像真实数据”，而业务规则判断仍然以技术方案为主。

## 验证方式

对生成后的 Markdown 文档运行：

```bash
python scripts/validate_test_data.py <Markdown文件路径>
```

脚本会检查：

- 是否为 `测试执行清单.md` 结构
- 是否至少包含一条 `TC-xxx`
- 是否支持“场景组 > TC > 小节”的嵌套结构
- 每条 `TC` 是否都包含 `前置条件 / 测试步骤 / 预期结果 / 输入数据 / 说明 / 待确认项`
- `输入数据` 是否按“表：”分块展示

当前只支持新版 `测试执行清单.md`，不再支持旧版表格稿或“基础记录池 + 用例引用”格式。
