# 测试执行清单 Skill

用于把已确认的 `结构化测试用例` 继续展开成可执行的 `测试执行清单`。

## 目标

这个 Skill 重点回答三件事：

1. 每条 `TC` 需要准备什么数据
2. 数据落在哪些表、依赖哪些关联
3. 哪些规则可以直接确定，哪些必须依赖技术方案

它不负责产出可执行 SQL，也不建议把结果写成抽象的数据池总表。

## 与 `test-case-generator` 的关系

推荐按“基线稿 + 执行稿”使用：

1. `test-case-generator` 先输出并确认 `结构化测试用例`
2. `test-data-generator` 再读取已确认用例、技术方案和 `reference-pack`
3. 最终输出 `<需求名称>_测试执行清单.md`

## 推荐输出结构

推荐正文结构：

1. `# <需求名称> 测试执行清单`
2. `## 输入材料`
3. `## 阅读导航`
4. `## 场景组X：<场景组名称>`
5. `### TC-xxx 测试功能点`

每条 `TC` 固定包含：

- 一张 5 列用例表：`编号 / 测试功能点 / 前置条件 / 测试步骤 / 预期结果`
- `输入数据`
- `说明`
- `待确认项`

补充约束：

- 默认继承场景地图中的场景分组名称与顺序
- `输入数据` 必须按表分块展示，不能跨表混排
- 每张表推荐使用 `**表：表名**` 作为块标签
- 每张表只保留当前 `TC` 需要的最小充分记录
- 保存候选版时默认按场景组分批写入同一目标文件；第一批先写文档头部、`输入材料`、`阅读导航` 和第一个场景组，后续批次再逐个追加剩余场景组
- 如果单个场景组仍然过大，再在该场景组内部按完整 `TC` 分批写入；不要把一个 `TC` 拆成多批
- 全部批次写完后，再统一运行校验脚本；不要在中间批次反复校验

## 建议输入

主输入：

1. 已确认的 `结构化测试用例`
2. 技术方案
3. `reference-pack` 路径
  - `tables/*.sql`
  - `table_samples/*.csv`
  - 如果是全库级 pack，建议同时提供 `table_index.csv`

补充输入：

- 接口说明
- 状态或枚举映射说明
- 历史修复 SQL 或迁移说明

## 技术方案的重要性

技术方案是核心输入，不只是补充材料。

以下场景如果缺技术方案，通常无法给出高置信度执行清单：

- 展示字段不是数据库原字段
- 状态值、标签、枚举需要映射
- 历史修复影响修复前后判断
- 权限规则影响操作入口和结果
- 日志展示与日志落表不一致
- 同一业务字段来自多表拼装或跨系统同步

`reference-pack` 只能帮助确认字段落点和样例形态，不能替代技术方案。

## `reference-pack` 的作用

这个 Skill 会结合：

- `references/data-design-rules.md`
- `references/data-coverage-checklist.md`
- `assets/test-data-template.md`
- `assets/test-execution-example.md`

其中 `table_samples/*.csv` 主要用来参考字段形态、取值风格和常见组合，不建议整张照抄进执行清单。

如果 `reference-pack` 是全库导出、包含很多表，推荐使用下面的读取约束：

- 先读技术方案，再读 `table_index.csv`，先按业务词、表名、表注释和关键字段缩圈
- 默认把全库 pack 当作“候选库”，不是默认整包通读的输入
- 首轮通常只展开 `3-10` 张候选表的 `tables/*.sql` 与 `table_samples/*.csv`
- 如果候选表仍然很多，应该先回到需求范围、接口字段或业务关键词继续收敛
- 输出测试执行清单时，建议显式说明“本次选了哪些表、为什么选它们”

## 验证方式

从仓库根目录运行：

```bash
python skills/test-data-generator/scripts/validate_test_data.py <Markdown文件路径>
```

如果你已经在当前目录 `skills/test-data-generator/` 下，也可以运行：

```bash
python scripts/validate_test_data.py <Markdown文件路径>
```

脚本会检查：

- 文档标题是否包含 `测试执行清单`
- 是否至少包含一条 `TC-xxx`
- 是否符合“场景组 > TC > 小节”的嵌套结构
- 每条 `TC` 是否都包含 5 列用例表、`输入数据`、`说明`、`待确认项`
- `输入数据` 是否按 `**表：表名**` 或兼容旧版表级标题分块展示

如果采用分批保存，推荐顺序是：

1. 第一批写入文档头部、`输入材料`、`阅读导航` 和第一个场景组
2. 后续批次按完整场景组追加
3. 若单个场景组仍然过大，再按完整 `TC` 批次追加
4. 全部批次完成后，再执行一次最终校验

推荐文件名是 `<需求名称>_测试执行清单.md`，但校验脚本检查的是文档标题和结构，不强制磁盘文件名固定为 `测试执行清单.md`。

当前交付规范以新版结构为准；校验脚本对部分旧式小节仍保留有限兼容，但不建议继续使用旧格式。
