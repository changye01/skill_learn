# 测试执行清单 Skill

一个与 `test-case-generator` 串联使用的 Skill，用于把“已确认测试用例”进一步转成“可直接执行的测试执行清单”。

## 目标

把“结构化测试用例 -> 测试执行清单”的过程单独拆出来，明确回答：

1. 每条 `TC` 执行时到底要准备哪些数据。
2. 这些数据落在哪些表、依赖哪些关联、验证哪些断言。
3. 哪些规则可以确定，哪些仍需技术方案补充确认。

默认目标是 `执行信息完整 + 测试数据可直接落到 TC 上下文中阅读`，而不是额外维护一份抽象的数据池总表，也不是直接产出可执行 SQL。

## 与 `test-case-generator` 的关系

推荐使用“基线稿 + 执行稿”模式：

1. `test-case-generator` 先输出并确认 `结构化测试用例.md`，负责定义 `TC`、前置条件、测试步骤、预期结果。
2. `test-data-generator` 再读取已确认用例和 `reference-pack`，补充测试数据、说明、待确认项，输出 `测试执行清单.md`。

这样可以把“测什么 / 怎么测”和“拿什么数据执行”分开管理，同时保证最终执行稿按 `TC` 阅读顺手。

## 推荐输出

正式输出统一为 `Markdown` 版 `测试执行清单.md`，推荐结构如下：

1. `# <需求名称> 测试执行清单`
2. `## 输入材料`
3. `## TC-xxx 测试功能点`

其中每条 `TC` 固定展开：

- `前置条件`
- `测试步骤`
- `预期结果`
- `测试数据`
- `说明`
- `待确认项`

约束说明：

- 不保留顶部总表
- 不再以“基础记录池”作为主阅读结构
- 同一条数据命中多个 `TC` 时，允许在多个 `TC` 下重复展示
- `测试数据` 必须按表分块展示，不能跨表混排
- 每张表只保留“关键断言字段 + 关联字段 + 必要上下文字段”的最小充分记录

## 工作流程

1. 确认已确认的 `结构化测试用例.md` 和 `reference-pack` 路径。
2. 判断是否存在派生字段、状态映射、历史修复、跨系统链路等复杂规则。
3. 若有复杂规则但缺少技术方案，先提示补充参考材料。
4. 逐条继承基线用例中的 `TC / 前置条件 / 步骤 / 预期`。
5. 读取 `tables/*.sql` 和 `table_samples/*.csv`，定位每条用例所需的表、字段和值形态。
6. 输出 Markdown `测试执行清单` 草稿。
7. 根据覆盖清单补漏，并把无法确认的信息留在对应 `TC` 的 `待确认项`。
8. 用户确认草稿方向后保存候选版 Markdown。
9. 运行校验脚本做 Markdown 结构校验。
10. 如校验失败，回改 Markdown 并重新校验。
11. 校验通过后，再将其视为确认版。

## 建议输入

### 主输入

1. 已确认的 `结构化测试用例.md`
2. `reference-pack`
   - `reference-packs/tables/*.sql`
   - `reference-packs/table_samples/*.csv`

### 补充输入

- 技术方案
- 接口说明
- 状态/枚举映射说明
- 历史修复 SQL 或迁移说明

## 验证方式

对生成后的 Markdown 文档运行：

```bash
python scripts/validate_test_data.py <Markdown文件路径>
```

脚本会检查：

- 是否为 `测试执行清单.md` 结构
- 是否至少包含一条 `TC-xxx`
- 每条 `TC` 是否都包含 `前置条件 / 测试步骤 / 预期结果 / 测试数据 / 说明 / 待确认项`
- `测试数据` 是否按 `#### 表：` 分块展示

当前只支持新版 `测试执行清单.md`，不再支持旧版表格稿或“基础记录池 + 用例引用”格式。
